## 🧱 一、 全局架构地图 (Monorepo 隔离规范)

本项目严格采用多包 (Monorepo/Packages) 架构。在生成任何代码时，你必须明确指出文件属于 `packages/` 还是主工程 `lib/`。

```text
/ (Project Root)
├── packages/           (🔥🔥🔥 跨应用高度复用中台 SDK 区)
│   ├── membership_sdk/ # 会员核心业务与通用接口 (绝不含具体App的UI)
│   ├── payment_sdk/    # 聚合支付中台 (微信/支付宝/苹果内购等纯逻辑)
│   └── ad_sdk/         # 广告推送与分发聚合引擎
│
└── lib/                (🚀 当前 App 主工程)
    ├── app/            (启动环境与全局注册中心) 
    ├── core/           (当前App的基础设施适配)
    ├── engine/         (⚠️ 核心枢纽：鉴权引擎、服务端配置同步) 
    ├── features/       (独立业务模块/页面，消费 SDK 数据)
    └── shared/         (全局UI组件)
[❌ 绝对禁区 1：反向依赖] > 绝对禁止在 packages/ 中的代码里 import 主工程 lib/ 下的任何文件！SDK 必须是纯净、无状态、可独立发布到其他 App 的。

⚙️ 二、 核心系统设计规范
1. SDK 中台原则 (会员/支付/广告)
纯逻辑原则：对外只暴露 API 与统一状态（例：PaymentEngine.pay(), MembershipManager.isVIP()）。

UI 剥离原则：SDK 内部不提供完整的复杂页面，只提供基础的通用弹窗（可选）或纯数据。具体的业务展示必须由对应 App 的 features/ 去实现并调用 SDK。

2. 服务端驱动权限 (Server-Driven Permissions)
统一管控：App 的所有会员权限控制，强制通过 engine/permission_engine.dart 统一管控。

同步机制：在 app/ 层启动时，必须调用权限引擎向服务器拉取最新的权限配置 JSON，并写入全局内存。

[❌ 绝对禁区 2：硬编码权限] 绝对禁止在 UI 层 (Widget) 或 Feature 内部写死 if (user.level == 1) 这种硬编码。必须通过 PermissionEngine.hasPermission('feature_key') 校验。

3. 状态管理与事件流 (唯一合法方式)
跨模块状态与事件通信【只能】通过全局 Config/EventBus：

读写状态：Config.set("membership.status", vip);

监听事件：Config.watch("ad.on_push_received", callback);

[❌ 绝对禁区 3：违规调用] 禁止 UI 组件直接调用持久化接口，UI 的变化只能通过监听 Config/状态引擎驱动。

🔌 三、 模块 (Feature) 标准结构
主工程的 features/xxx/ 必须包含且仅包含以下结构（强制统一）：

ui/ ：纯粹视图（通过鉴权控制按钮隐藏/置灰）。

controller/ ：逻辑控制（调用鉴权引擎与 SDK 中台）。

service/ ：模块专属接口。

model/ ：模块专属数据模型。

register.dart ：模块注册中心。

📝 四、 编码与注释规范
强制中文注释：所有函数、类、核心变量以及传入参数，必须写上清晰的中文备注，说明其功能和作用。

同步修改：如果后续重构或修改了某段逻辑，其对应的注释必须同步更新，决不允许注释与代码脱节。

## 🎨 七、 UI 组件与设计系统规范 (严格执行)

为了保证后期支持一键切换主题（深色/浅色/自定义配色、背景透明度）以及多语言切换，你在编写任何 UI 代码时，必须遵守以下铁律：

[❌ 绝对禁区 4：硬编码 UI 值] 绝对禁止在 UI 代码中出现任何硬编码的颜色、字体大小、间距数字和明文文本！
- ❌ 禁止：`color: Colors.blue` / `color: Color(0xFF000000)`
- ❌ 禁止：`fontSize: 16.0` / `padding: EdgeInsets.all(12)`
- ❌ 禁止：`Text("确认登录")`

### 1. 主题与配色 (Theme & Colors)
所有的颜色、透明度、渐变，必须通过全局 Theme 引擎获取。
- ✅ 颜色获取规范：使用 `Theme.of(context).colorScheme.xxx` 或自定义主题扩展 `Theme.of(context).extension<AppColors>()!.xxx`。
- ✅ 透明度处理：如果需要半透明背景，必须基于主题色调用 `.withOpacity(0.5)`，而不是重新定义颜色。
- 💡 (注：在生成组件前，先问自己：这个颜色是属于 primary、secondary 还是 surface 背景色？)

### 2. 排版与尺寸 (Typography & Spacing)
所有的字体样式和间距，必须使用全局预设。
- ✅ 字体规范：使用 `Theme.of(context).textTheme.bodyMedium` 等标准文字样式。
- ✅ 间距规范：使用项目中已定义的全局常量（如 `AppSpacing.md`）或相对屏幕的百分比，适配跨端需求。

### 3. 国际化与多语言 (i18n & l10n)
所有的展示文本必须走多语言配置通道。
- ✅ 文本规范：必须使用多语言调用方法（例如：`S.of(context).login_confirm` 或 `context.tr('login_confirm')`，具体依据项目现有方案）。
- 🚧 自动补充要求：如果你在 UI 中加入了一个新的文本节点，你必须在输出代码后，主动提醒我：“请将以下键值对添加到语言翻译文件中：`'login_confirm': '确认登录'`”。

### 4. 样式剥离 (Widget 解耦)
如果一个复杂的交互组件（如带有多重状态的卡片）超过 100 行，必须将其 UI 样式部分提取为一个独立的 `StatelessWidget`，并将所有颜色、圆角、背景作为构造函数的参数（或使用默认 Theme），以便未来单独抽离成 UI 库。
5.语言切换必须涉及到app所有的设置，除了用户手动输入的文本，以及书籍的内容。
