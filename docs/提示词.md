## 🧱 一、 全局架构地图 (Monorepo 隔离规范)

本项目严格采用多包 (Monorepo/Packages) 架构。在生成任何代码时，你必须明确指出文件属于 `packages/` 还是主工程 `lib/`。


/ (Project Root)
├── packages/           (🔥🔥🔥 跨应用高度复用中台 SDK 区)
│   ├── membership_sdk/ # 会员核心业务与通用接口 (绝不含具体App的UI)
│   ├── payment_sdk/    # 聚合支付中台 (微信/支付宝/苹果内购等纯逻辑)
│   └── ad_sdk/         # 广告推送与分发聚合引擎
│
└── lib/                (🚀 当前 App 主工程)
    ├── app/            (启动环境与全局注册中心) 
    ├── core/           (当前App的基础设施适配)
    ├── engine/         (⚠️ 核心枢纽：鉴权引擎、服务端配置同步) 
    ├── features/       (独立业务模块/页面，消费 SDK 数据)
    └── shared/         (全局UI组件) 

> **[❌ 绝对禁区 1：反向依赖]** 绝对禁止在 `packages/` 中的代码里 import 主工程 `lib/` 下的任何文件！SDK 必须是纯净、无状态、可独立发布到其他 App 的。

## ⚙️ 二、 核心系统设计规范

### 1. SDK 中台原则 (会员/支付/广告)

- **纯逻辑原则**：对外只暴露 API 与统一状态（例：`PaymentEngine.pay()`, `MembershipManager.isVIP()`）。
    
- **UI 剥离原则**：SDK 内部不提供完整的复杂页面，只提供基础的通用弹窗（可选）或纯数据。具体的业务展示必须由对应 App 的 `features/` 去实现并调用 SDK。
    

### 2. 服务端驱动权限 (Server-Driven Permissions)

- **统一管控**：App 的所有会员权限控制，强制通过 `engine/permission_engine.dart` 统一管控。
    
- **同步机制**：在 `app/` 层启动时，必须调用权限引擎向服务器拉取最新的权限配置 JSON，并写入全局内存。
    

> **[❌ 绝对禁区 2：硬编码权限]** 绝对禁止在 UI 层 (Widget) 或 Feature 内部写死 `if (user.level == 1)` 这种硬编码。必须通过 `PermissionEngine.hasPermission('feature_key')` 校验。

### 3. 状态管理与事件流

跨模块状态与事件通信**只能**通过全局 Config/EventBus：

- **读写状态**：`Config.set("membership.status", vip);`
    
- **监听事件**：`Config.watch("ad.on_push_received", callback);`
    

> **[❌ 绝对禁区 3：违规调用]** 禁止 UI 组件直接调用持久化接口，UI 的变化只能通过监听 Config/状态引擎驱动。

## 🎨 三、 UI 组件与设计系统规范 (严格执行)

为了保证后期支持一键切换主题（配色/透明度）、多语言无缝切换以及云端字体动态下载，编写 UI 代码必须遵守以下铁律：

> **[❌ 绝对禁区 4：硬编码 UI 值]** 绝对禁止在 UI 代码中出现任何硬编码的颜色、字体大小、间距数字和明文文本！
> 
> - ❌ 禁止：`color: Colors.blue` / `color: Color(0xFF000000)`
>     
> - ❌ 禁止：`fontSize: 16.0` / `padding: EdgeInsets.all(12)`
>     
> - ❌ 禁止：`Text("确认登录")`
>     

> **[❌ 绝对禁区 5：私自声明字体]** 绝对禁止在任何 UI 组件 (Widget) 中使用 `TextStyle(fontFamily: 'xxx')` 来硬编码字体名称！所有的字体定义必须交由全局 Theme 引擎接管。

### 1. 主题与配色 (Theme & Colors)
### 1. 主题与动态配色引擎 (Dynamic Theme Engine)
本项目支持用户自定义主题（深浅模式、自定义主色调、背景透明度等），所有的动态生成与持久化逻辑必须高度集中。

> **[❌ 绝对禁区 6：局部状态管理主题]** 绝对禁止在任何 Feature 的页面内使用 `setState` 或局部状态来直接控制颜色的变化！颜色和主题的变更必须上浮到全局 App 级别。

* **UI 层读取规范 (Read)**：
  * 只能使用 `Theme.of(context).colorScheme.xxx` 或 `Theme.of(context).extension<AppColors>()!.xxx`。
  * **透明度绑定**：如果用户调整了全局“背景透明度”，该参数必须注入到 Theme 引擎中统一计算，UI 组件只能调用 `.withOpacity(Theme.of(context).extension<AppColors>()!.bgOpacity)`。

* **引擎层修改规范 (Write & React) (仅供修改主题功能时参考)**：
  * **唯一入口**：所有的主题变更操作（如用户在设置页选择了一个新颜色），必须通过全局统一接口调用，例如 `ThemeEngine.setPrimaryColor(newColor)`。
  * **动态生成与持久化**：`core/theme/theme_engine.dart` 接收到新颜色后，必须负责：
    1. 将用户选择的 Hex 颜色值持久化存储（本地缓存或服务端）。
    2. 基于新颜色，动态推导出完整的 `ColorScheme`（使用 `ColorScheme.fromSeed` 等原生方法）。
    3. 触发全局状态更新，通知顶层 `MaterialApp/CupertinoApp` 重新构建 Theme。
  * **UI 无感知**：底层的 Feature 页面不关心颜色是怎么存的、怎么生成的，它们只需要在顶层重新构建时自然响应最新的 `Theme.of(context)` 即可。
    

### 2. 字体与排版动态化 (Dynamic Typography)

本项目需支持“多字体自由切换”及“云端字体动态下载”，字体应用必须 100% 抽离。

- **统一调用规范**：UI 层只能使用 `Theme.of(context).textTheme` 下的语义化样式（如 `titleLarge`, `bodyMedium`）。如果需要改变字重或颜色，使用 `copyWith()` 方法，但**绝对不允许在 `copyWith` 中传入 `fontFamily`**。
    
- **底层架构约束**：所有的字体加载逻辑、内置双字体切换、网络下载逻辑，必须且只能封装在 `core/theme/font_manager.dart` (主工程) 或 `packages/shared/ui/` 中。业务模块对当前使用的字体必须保持**完全无感知 (Zero Awareness)**。
    
- **间距规范**：使用项目中已定义的全局常量（如 `AppSpacing.md`）或相对屏幕的百分比，适配跨端需求。
    

### 3. 国际化与多语言 (i18n & l10n)

- **全局覆盖原则**：语言切换必须涉及到 App 所有的设置和展示内容（**除用户手动输入的文本，以及书籍的核心内容本身外**）。
    
- **文本规范**：必须使用多语言调用方法（例如：`S.of(context).login_confirm` 或 `context.tr('login_confirm')`）。
    
- **自动补充要求**：如果在 UI 中加入了新的文本节点，必须在输出代码后，主动提醒我：“请将以下键值对添加到语言翻译文件中：`'login_confirm': '确认登录'`”。
    

### 4. 样式剥离 (Widget 解耦)

- 如果一个复杂的交互组件（如带有多重状态的卡片）超过 100 行，必须将其 UI 样式部分提取为一个独立的 `StatelessWidget`，并将所有颜色、圆角、背景作为构造函数的参数（或使用默认 Theme），以便未来单独抽离成 UI 库。
    

## 🔌 四、 模块 (Feature) 标准结构

主工程的 `features/xxx/` 必须包含且仅包含以下结构（强制统一）：

- `ui/`：纯粹视图（通过鉴权控制按钮隐藏/置灰）。
    
- `controller/`：逻辑控制（调用鉴权引擎与 SDK 中台）。
    
- `service/`：模块专属接口。
    
- `model/`：模块专属数据模型。
    
- `register.dart`：模块注册中心。
    

## 📝 五、 编码、依赖与注释规范

### 1. 插件与依赖选择 (跨平台优先)

- **性能优先，多端兼容**：在引入任何第三方包时，优先使用性能最强（如 FFI 底层实现），且明确支持 Windows、iOS、Android、macOS 全平台通用的插件。
    

### 2. 注释规范

- **强制中文注释**：所有函数、类、核心变量以及传入参数，必须写上清晰的中文备注，说明其功能和作用。
    
- **同步修改**：如果后续重构或修改了某段逻辑，其对应的注释必须同步更新，决不允许注释与代码脱节。
    

## 🤖 六、 AI 代理强制执行工作流 (SOP)

每次接收到用户的开发需求时，你必须严格按照以下 3 步执行，不可跳过：

### 步骤 1：响应确认 (必须在写任何代码前输出)

简短地向用户确认你的理解：

### 步骤 2：架构越界拦截机制 (Safety Check)

- **必须暂停输出代码！** 先向用户输出：【跨模块修改原因】、【影响范围】和【修改清单】。获得用户明确同意后方可生成代码。
    

### 步骤 3：按序生成代码

一旦确认无误，必须按以下顺序输出代码：

1. 中台接口定义 (如果在 `packages/`，先写无 UI 的核心交互)。
2. 服务端权限映射 (定义配置 Key)。
3. Feature 结构 (依次输出 Model -> Service -> Controller -> UI)。
4. 注册与鉴权接入 (输出 `register.dart` 及权限拦截逻辑)。
