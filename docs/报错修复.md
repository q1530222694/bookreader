# 🛠️ 报错修复协议 (Bug Fix SOP)

我遇到了一个代码报错/异常，请帮我排查并修复。
在进行修复时，你必须进入【外科手术模式】，严格遵守以下纪律，绝不能为了消除报错而破坏我的代码底座：

## 🚨 核心禁区 (不可逾越)
> **[❌ 绝对禁区 1：篡改业务逻辑]** 绝对禁止修改我原有的业务流程、条件判断逻辑 (if/else)、状态流转顺序。你的任务仅仅是“修好这根断掉的水管”，而不是“重新设计整个供水系统”。
> **[❌ 绝对禁区 2：抹除代码痕迹]** 绝对禁止在重写代码时，删除我原有的中文注释、TODO 标记、日志打印 (Log/print) 以及未报错的其他字段。
> **[❌ 绝对禁区 3：粗暴掩盖错误]** 绝对禁止在没有查明根本原因的情况下，通过大面积嵌套 `try-catch`，或随意使用强制解包 (`!`)、随意将非空类型改为可空类型 (`?`) 来强行压制报错。

## 🎯 修复原则
1. **最小化干预 (Surgical Precision)**：哪里报错改哪里。如果只需要改 1 行代码，绝对不要重写我整个 100 行的类或方法。
2. **遵守既有架构**：修复方案必须 100% 遵守项目当前的 `AI_RULES.md`（如：使用 Theme 引擎处理 UI、通过全局 Config 传递状态、保持中台 SDK 纯净）。不得为了方便引入局部状态 (`setState`) 或硬编码。

## 🤖 强制执行工作流 (请按以下 3 步输出)

**步骤 1：根因诊断 (Root Cause Analysis)**
用一两句话简明扼要地指出：到底是什么导致了这个报错？（是空指针、类型不匹配、平台插件缺失、还是生命周期错乱？）

**步骤 2：最小化修复方案 (Minimal Fix Proposal)**
说明你打算怎么改，确保明确告诉我这个修改**没有**影响上下文的业务逻辑。

**步骤 3：输出修复代码 (Targeted Code)**
只输出**修改过的部分**（函数或局部代码块），并带上周围几行上下文，方便我定位替换。不要一言不合就输出整个几百行的文件内容！
