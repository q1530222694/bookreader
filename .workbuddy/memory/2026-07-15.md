## 修复：改 pdfrx 后 macOS 无法启动（BUILD FAILED）
- **现象**：`flutter run -d macos` 报 ① CocoaPods 残留警告；② 致命错误「flutter-onnxruntime requires minimum platform version 14.0 but this target supports 10.15」→ BUILD FAILED。
- **根因**：引入 `flutter_onnxruntime`（OCR 阶段0，全平台 FFI）后其 Swift Package 要求 macOS ≥ 14.0，而 `Runner` 的 `MACOSX_DEPLOYMENT_TARGET` 仍为 10.15；同时早期 `pod install` 残留的 CocoaPods 集成与 Flutter 当前的 SwiftPM 插件机制并存。
- **修复动作**（仅改 `macos/` 原生构建配置，未触碰 `lib/`/`packages/`）：
  1. `cd macos && pod deintegrate`，并删除 `macos/Podfile`、`macos/Podfile.lock`、`macos/Pods/`。
  2. 去掉 `macos/Flutter/Flutter-Debug.xcconfig` 与 `Flutter-Release.xcconfig` 首行的 `#include? "Pods/..."`。
  3. `macos/Runner.xcodeproj/project.pbxproj` 三处 `MACOSX_DEPLOYMENT_TARGET = 10.15` → `14.0`（PBXProject 级 Debug/Profile/Release）。
  4. 清理 `macos/Runner.xcworkspace/contents.xcworkspacedata` 中残留的 `Pods/Pods.xcodeproj` 引用。
  5. `flutter pub get` 成功，Podfile 不再被重新生成（全部原生插件支持 SwiftPM）。
- **原理**：Flutter 在 `flutter run/build` 时由 `SwiftPackageManager.updateMinimumDeployment` 读取 `MACOSX_DEPLOYMENT_TARGET`(现已 14.0)，把自动生成的 `FlutterGeneratedPluginSwiftPackage/Package.swift` 平台从默认 10.15 提升为 14.0，从而满足 onnxruntime 要求。
- **副作用**：App 最低支持系统抬到 **macOS 14.0（Sonoma）**，是引入 flutter_onnxruntime 的必然后果。
- **验证限制**：本 agent 沙箱环境的 OS 层禁止 `sandbox-exec`（Xcode 解析 SwiftPM 依赖时调用），导致本地 `flutter build macos` 无法跑通（报 `posix_spawn error: Operation not permitted`），此为环境限制而非配置问题，用户真实 Mac 上可正常构建。

## 修复（根因）：书籍点开即被裁切、双排更严重 —— pdfrx render 参数误用
- **现象**：用户确认自动裁切关闭，但进书即被切掉一部分，双排（双页）裁切更明显。
- **根因（已读 pdfrx 1.3.5 源码 `lib/src/pdfium/pdfrx_pdfium.dart:607` 确认）**：`PdfPage.render` 在 `fullWidth/fullHeight` 缺省时 `??=` 回退为**页面原生 pt 尺寸**（如 595×842）；旧「无裁切」分支调 `page.render(width: fullW, height: fullH)` 漏传 `fullWidth/fullHeight`，于是位图尺寸(width/height，单页约 1200、双页约 591) **小于** 原生页 → PDFium 把整页按原生尺寸渲染进更小位图，**右/下边缘溢出被静默裁掉**；双页 `targetWidth=(pageW-6)/2` 更小→裁切比例更大→「双排更严重」。这是**真正的根因**（此前 10 项修复里⑧「随自动裁切重算」判断不对症）。
- **修复（仅 `pdf_render_service.dart`）**：
  1. 无裁切分支改 `page.render(fullWidth: fullW, fullHeight: fullH)`（不传 x/y/width/height）→ `width ??= fullWidth`、`height ??= fullHeight` → 位图尺寸==整页渲染尺寸，零裁切。
  2. 兜底分支（包围盒退化整页）同样改 `fullWidth/fullHeight`。
  3. 自动裁切探针 `computeCropFractions` 原 `render(width: cropScanWidth, height: probeH)` 漏传 `fullWidth/fullHeight`，探针只扫到整页左上角、包围盒偏小（开自动裁切仍误切右侧）→ 改 `render(fullWidth: cropScanWidth, fullHeight: probeH)` 成真正全页探针。
- **防御性加固（早于本修复，已就位）**：`pdf_custom_view` 单页 `FittedBox(BoxFit.contain)` 包裹 `RawImage`；双页 `Row` 外层 `ClipRect(clipBehavior: Clip.none)`。与 render 根因修复互补。
- **坑（牢记）**：用 pdfrx 低层 `PdfPage.render` 时，若要渲染「整页」必须显式传 `fullWidth/fullHeight`；只传 `width/height` 会被当成「位图尺寸」而 `fullWidth/fullHeight` 回退原生页尺寸，页面大时即裁切。
- **验证**：`flutter analyze` 两个文件 → 0 error。沙箱禁 `sandbox-exec` 无法本地 run，需用户本机实测：关闭自动裁切进任意 PDF，确认单页/双页边缘完整。
- **文档**：已追加 `docs/CHANGELOG.md` 根因条目 + 更新 `docs/不同文件的依赖关系.md` 的 `pdf_render_service.dart` 条目。
